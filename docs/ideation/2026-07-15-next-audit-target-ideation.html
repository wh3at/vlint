Grounding Context

vlint は、有限の宣言済み UI target を実ブラウザで開き、画像や LLM を使わず DOM geometry 契約を検査する CLI。現行監査は tab-label-single-line のみ。PRD は文字切れ、要素衝突、親領域突出、ページ全体 overflow、余白・alignment、tap 領域を将来 rule として明示している。

v0.2.0 では MacBook Air と iPhone 17 の target × device 監査が入り、同一 URL が desktop では clean、mobile では violation になる契約が acceptance test で固定された。一方、設定型、評価結果、schema、dispatcher はまだ tab 専用。二本目の rule は利用者価値だけでなく、汎用 rule 契約を正しく作る楔になる。

外部先行例では Polypane が水平 overflow の原因特定を製品機能として提供。ReDECHECK/VISER 系研究は DOM-only の広範な collision 検出で誤検知が問題になることを示す。したがって、明確な global symptom と element-scoped attribution を組み合わせる案を優先した。

Top pick

Horizontal page overflow + source trace

document.documentElement.scrollWidth > window.innerWidth でページ全体の水平 overflow を検出し、右端超過が最大の要素とその祖先 chain を追跡する。診断は超過量、原因要素 geometry、一意 locator、関連する min-width / fixed width / white-space を返す。overflow-x:auto|scroll に閉じ込められた carousel、table、code block は intentional boundary として除外する。

Axis
マルチデバイス価値・AI向け原因診断
Basis
direct: vlint-prd.md:5,177,202、tests/acceptance/vlint.test.ts:816-844。external: Polypane の horizontal overflow 原因検出。reasoned: root overflow は意図されにくく、原因要素へ帰属できれば修正可能性が高い。
Rationale
desktop で見逃し、402px で顕在化する代表的 responsive failure。selector 無しで発見でき、agent が max-width、min-width:0、wrap、局所 scroll のどこを直すべきか判断できる。
Downsides
意図的な横 scroll を boundary 判定しないと誤検知する。現行の Violation.lineCount / labelsInspected と page-vs-element scope を汎用化する必要がある。

Click occlusion via hit testing

button、link、tab、input などの interactive element の中心点で document.elementFromPoint() を実行し、topmost element が対象自身または子孫でなければ occlusion と判定する。generic rectangle collision ではなく「実際に click を奪う重なり」へ監査対象を再定義する。

Axis
決定論的検出精度・利用者価値
Basis
direct: PRD の「要素衝突」候補と既存 locator pipeline。reasoned: hit-test は paint order と pointer-events を反映し、badge や装飾の単純 overlap を避ける。
Rationale
ユーザーが押せないという実害を直接測るため、collision scan より診断価値が高い。
Downsides
中心以外だけが塞がれる部分 occlusion は取りこぼす。複数 sample point は精度とコストの設計が必要。

Tap target minimum size and spacing

semantic interactive elements を自動発見し、rendered hit area と隣接 target までの距離を CSS pixel で監査する。WCAG 2.5.8 の 24×24 CSS px を基準にし、より厳しい 44×44 profile は設定で選択可能にする。

Axis
利用者価値・決定論的検出精度
Basis
external: WCAG 2.5.8 と axe-core target-size rule。direct: device profile の hasTouch と既存 geometry/locator。
Rationale
明確な標準と成熟した例外設計があり、mobile geometry の価値が高い。
Downsides
axe-core と機能重複しやすい。文中 link、UA default control、spacing exception の仕様移植が必要。

Parent overflow / protrusion

child rect が ancestor content box を超える量を測り、局所的な突出を検出する。root document の scrollWidth が変わらない overflow:hidden や clip の内側も対象にできる。

Axis
設定・結果契約適合性
Basis
direct: PRD の「親領域からの突出」、既存 getBoundingClientRect() と computed-style 取得。reasoned: #1 と同じ parent-child relation primitive を共有しつつ blind spot を補完する。
Rationale
#1 の次に追加すると geometry relation と intent boundary を償却できる。
Downsides
overflow:visible、transform、popover、decorative spill は意図的な場合が多い。全 ancestor を対象にすると noisy。
Confidence
83% · High

Silent text truncation

scrollWidth > clientWidth または text fragment と visible box の包含関係から文字切れを検出する。明示的な text-overflow:ellipsis と line clamp は意図の proxy として clean、ellipsis 無しの clip を優先的に violation とする。

Axis
AI向け原因診断
Basis
direct: PRD の「文字切れ」、既存 Range fragment extractor。reasoned: 現行 rule と同じ text geometry を別 invariant へ再利用できる。
Rationale
wrap しないまま内容が消える、現行 rule の明確な blind spot を埋める。
Downsides
ellipsis 無しの clip も意図的な場合があり、computed style だけでは完全な intent 判定にならない。

6

Generic semantic label single-line

既存の line clustering をそのまま使い、button、navigation item、segmented control など selector-scoped な semantic label へ一般化する。最小リスクで二本目 rule の dispatch と type union を導入する案。

Axis
設定・結果契約適合性
Basis
direct: clusterLineCount は候補 role から独立し、現 schema の type literal が一般化点を明示している。
Rationale
最も安全な architectural wedge。ただし利用者価値は overflow より小さい。
Downsides
lineCount shape が同じため、generic violation metric の難題を三本目へ先送りしやすい。

90% · High
